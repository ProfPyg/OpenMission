1. OVERVIEW
================================================================================

OpenMission is a browser-based tool for preliminary mission analysis of
Sun-Synchronous Orbit (SSO) constellations for Earth Observation (EO)
applications. It computes coverage and revisit metrics over a user-defined
Region of Interest (ROI) and access windows for one or more ground stations.

The tool runs entirely in the browser with no server or network dependencies.
All computations are performed client-side using a J2-perturbed analytical
propagator. Geographic border data is hardcoded into the application.

Primary outputs:
  - Mean and maximum revisit time over the ROI
  - ROI coverage percentage (fraction of grid points accessed at least once)
  - Per-ground-station: number of passes, access ratio, mean/max revisit,
    average pass duration

OpenMission is intended for early trade-space exploration. Results should be
verified against higher-fidelity tools (STK, MATLAB Aerospace Toolbox) before
use in final design.

2. INPUTS
================================================================================

--------------------------------------------------------------------------------
2.1  Constellation
--------------------------------------------------------------------------------

  Parameter           Input     Description
  ---------           -----     -----------
  Orbital planes      User      Number of orbital planes P
  Satellites/plane    User      Number of satellites per plane S
  Total satellites    Derived   N = P x S

Satellites within each plane are equally spaced in mean anomaly:

    M0(s) = s * (360 / S)    [deg],    s = 0, 1, ..., S-1

Planes are equally spaced in RAAN:

    RAAN(p) = RAAN0 + p * (360 / P)    [deg],    p = 0, 1, ..., P-1

--------------------------------------------------------------------------------
2.2  Classical Orbital Elements (COEs)
--------------------------------------------------------------------------------

  Element             Symbol    Input     Value / Formula
  -------             ------    -----     ---------------
  Semi-major axis     a         Derived   a = Re + h  [km],  Re = 6378.137 km
  Eccentricity        e         Locked    0.001 (near-circular)
  Inclination         i         Auto/Man  See Section 2.3
  RAAN (plane 1)      RAAN0     User      Editable, linked to LTAN
  LTAN (plane 1)      LTAN      User      Editable, linked to RAAN0
  Arg. of perigee     omega     Locked    0 deg (undefined at e ~ 0)
  Mean anomaly        M0        Locked    0 deg for first satellite of plane 1

Eccentricity is locked at 0.001 to maintain a near-circular orbit while
avoiding numerical issues in the Kepler equation solver at exactly e = 0.
The SSO inclination formula uses e = 0, consistent with the reference script.

RAAN and LTAN are live-linked. Editing either field updates the other
automatically using the Sun's right ascension at the simulation epoch:

    RAAN  = mod( alpha_s + 15 * (LTAN - 12),  360 )    [deg]
    LTAN  = mod( 12 + (RAAN - alpha_s) / 15,   24 )    [h]

where alpha_s is computed from Vallado (2013), Section 5.1, Eq. 5-4 to 5-7.

--------------------------------------------------------------------------------
2.3  Inclination — SSO Auto or Manual
--------------------------------------------------------------------------------

In SSO Auto mode, the inclination is computed from the altitude to satisfy
the sun-synchronous condition. The required RAAN precession rate is:

    omega_dot = 360 / 365.25 / 86400 * pi/180    [rad/s]  (~0.9856 deg/day)

Setting the J2 secular RAAN drift equal to omega_dot and solving for i
(Vallado, 2013, Eq. 9-41, with e = 0):

    cos(i) = - omega_dot / [ (3/2) * n * J2 * (Re/a)^2 ]

    i = acos( cos(i) )    [deg]

where n = sqrt(mu/a^3) is the mean motion and mu = 398600.4418 km^3/s^2.

In Manual mode, the user enters any inclination directly.

--------------------------------------------------------------------------------
2.4  Simulation Window
--------------------------------------------------------------------------------

  Parameter           Input     Description
  ---------           -----     -----------
  Start date          User      UTC calendar date
  Duration            User      1, 3, 7, 10, or 30 days
  Sample step         User      30 s (fine), 60 s (standard), 120 s (fast)

The propagator is evaluated at every sample step for all satellites.
Total computation steps = (duration in seconds) / (sample step).

--------------------------------------------------------------------------------
2.5  Payload Constraint
--------------------------------------------------------------------------------

For ROI access, two simultaneous conditions must be satisfied:

    Condition 1:  elevation angle  >=  min elevation    [deg]
    Condition 2:  off-nadir angle  <=  max off-nadir    [deg]

The off-nadir angle theta is the angle between the satellite nadir vector
and the line of sight to the target ground point (Wertz, 2012, Section 9.1):

    theta = arccos( r_sat_unit . (-delta_r / |delta_r|) )

where r_sat_unit is the unit vector from Earth centre to the satellite
and delta_r is the vector from the satellite to the ground point.

This constraint represents the optical payload's maximum pointing range from
nadir. No camera-specific parameters are required. Typical values for EO
optical payloads range from 20 deg to 45 deg.

Ground station access uses the elevation constraint only.

--------------------------------------------------------------------------------
2.6  Ground Stations
--------------------------------------------------------------------------------

The user defines ground stations by:

  - Name
  - Latitude  [deg N]
  - Longitude [deg E]
  - Altitude  [m]
  - Minimum elevation angle  [deg]

Any number of stations can be added or removed. Two stations are pre-loaded:

  Name              Lat [deg N]   Lon [deg E]   Alt [m]   Min El [deg]
  ----              -----------   -----------   -------   ------------
  Helmos NOA        37.9856       22.1983       2340      30
  UoA Psachna       38.5697       23.6486       80         5

The minimum elevation angle defines the lowest elevation above the local
horizon at which the satellite is considered visible from that station.

--------------------------------------------------------------------------------
2.7  Region of Interest (ROI)
--------------------------------------------------------------------------------

The user selects a country from the list of 44 supported European countries.
OpenMission generates a uniform rectangular grid over the country bounding box
and retains only the points lying inside the country border, using the
ray-casting point-in-polygon algorithm (Shimrat, 1962):

    Grid point (lat, lon) is inside the border if a horizontal ray cast
    eastward from that point crosses the border polygon an odd number of times.

The grid density is controlled by the user via rows and columns parameters.
Denser grids provide more representative statistics at the cost of longer
computation time.

Country borders are from Natural Earth 1:110m (public domain). The data is
hardcoded into the tool — no network request is made at runtime.

Supported countries (44):
  Albania, Armenia, Austria, Azerbaijan, Belarus, Belgium,
  Bosnia and Herzegovina, Bulgaria, Croatia, Cyprus, Czechia,
  Denmark, Estonia, Finland, France, Georgia, Germany, Greece,
  Hungary, Iceland, Ireland, Italy, Kosovo, Latvia, Lithuania,
  Luxembourg, Moldova, Montenegro, Netherlands, North Macedonia,
  Norway, Poland, Portugal, Romania, Russia, Serbia, Slovakia,
  Slovenia, Spain, Sweden, Switzerland, Turkey, Ukraine,
  United Kingdom

  Notes:
  - Russia: European portion only (lon < 65 deg E)
  - France: metropolitan only (overseas territories excluded)
  - Denmark, Norway: mainland and main islands only

3. PROPAGATOR
================================================================================

OpenMission uses a first-order analytical J2 propagator based on secular
perturbation theory. The satellite state is propagated from initial Keplerian
elements at each discrete time step.

--------------------------------------------------------------------------------
3.1  J2 Secular Perturbations
--------------------------------------------------------------------------------

The following secular rates are applied at each timestep
(Vallado, 2013, Eq. 9-41 and 9-42):

    dΩ/dt = -(3/2) · n · J2 · (Re/p)² · cos(i)          [rad/s]
    dω/dt =  (3/4) · n · J2 · (Re/p)² · (4 - 5·sin²i)   [rad/s]

where:
    n = sqrt(mu / a³)     mean motion [rad/s]
    p = a · (1 - e²)      semi-latus rectum [km]

Mean anomaly advances at the unperturbed mean motion:
    dM/dt = n

At each timestep t, the updated angles are:
    Ω(t) = Ω₀ + dΩ/dt · t
    ω(t) = ω₀ + dω/dt · t
    M(t) = M₀ + n · t

--------------------------------------------------------------------------------
3.2  Kepler's Equation
--------------------------------------------------------------------------------

At each timestep, Kepler's equation is solved for the eccentric anomaly E
(Vallado, 2013, §2.2, Eq. 2-4):

    M = E - e · sin(E)

Solved via Newton-Raphson iteration (8 iterations, sufficient for e = 0.001).
True anomaly and orbital radius follow from E.

--------------------------------------------------------------------------------
3.3  ECI Position
--------------------------------------------------------------------------------

The position vector in ECI is computed via the 3-1-3 Euler rotation
(Vallado, 2013, §2.6, Eq. 2-38):

    r_ECI = R_z(-Ω) · R_x(-i) · R_z(-ω) · r_perifocal

The propagator is fully vectorised — the full time array is passed at once,
returning an (N, 3) array of ECI positions for all N timesteps in a single
NumPy call.

--------------------------------------------------------------------------------
3.4  GMST Optimisation
--------------------------------------------------------------------------------

Greenwich Mean Sidereal Time is computed at the epoch using the full Vallado
polynomial (Eq. 3-45), then advanced linearly:

    GMST(t) = GMST₀ + ω_E · t

where ω_E = 7.2921150e-5 rad/s (Earth rotation rate).

This avoids calling the full polynomial at every timestep.
Error vs full polynomial: < 0.001° over 30 days.

================================================================================
4. ACCESS AND REVISIT COMPUTATION
================================================================================

--------------------------------------------------------------------------------
4.1  Elevation Angle
--------------------------------------------------------------------------------

At each timestep, the elevation angle of the satellite as seen from each
ground point is computed (Vallado, 2013, §4.4, Eq. 4-4):

    el = arcsin( (r_sat - r_gs) · n̂_gs / |r_sat - r_gs| )

where n̂_gs = r_gs / |r_gs| is the local zenith unit vector, and r_gs is the
ground station position in ECI computed via:

    r_gs = R_z(-GMST) · r_gs_ECEF

--------------------------------------------------------------------------------
4.2  Access Condition
--------------------------------------------------------------------------------

For ground stations:
    Visible if:  el >= min_elevation    [deg]

For ROI points:
    Visible if:  el >= (90 - max_off_nadir)    [deg]

which is equivalent to:  off-nadir angle <= max_off_nadir

The off-nadir angle θ is the angle between the satellite nadir vector and
the line of sight to the ground point (Wertz, 2012, §9.1):

    θ = arccos( r̂_sat · (-Δr / |Δr|) )

Since el + θ = 90° for a spherical Earth, only the elevation check is
computed in the inner loop for efficiency.

--------------------------------------------------------------------------------
4.3  Window Detection and Merging
--------------------------------------------------------------------------------

For each ground point, a boolean visibility array V(t) is computed across
all timesteps and all satellites. Access windows are detected by finding
0→1 and 1→0 transitions in V(t).

Windows from all satellites for the same ground point are collected and
merged using the direct port of the MATLAB merge loop from VLEOExample.m:

    currentStart = windows[0].start
    currentEnd   = windows[0].end
    for each subsequent window:
        if window.start <= currentEnd:
            currentEnd = max(currentEnd, window.end)
        else:
            save [currentStart, currentEnd]
            currentStart = window.start
            currentEnd   = window.end
    save final [currentStart, currentEnd]

--------------------------------------------------------------------------------
4.4  Revisit Metrics
--------------------------------------------------------------------------------

From the merged windows, revisit gaps are computed as:

    gap(i) = start(i+1) - end(i)    for i = 1, ..., N-1

Metrics per ground point:
    mean_revisit  = mean(gaps)
    max_revisit   = max(gaps)
    access_ratio  = sum(window durations) / simulation_duration × 100 %

ROI aggregate metrics:
    roi_mean_revisit  = mean of mean_revisit over all ROI points
    roi_max_revisit   = max of max_revisit over all ROI points
    roi_coverage      = fraction of ROI points with at least one access × 100 %

--------------------------------------------------------------------------------
4.5  Time Step Quantisation Error
--------------------------------------------------------------------------------

Access window boundaries are detected at the sample step resolution.
A sample step of dt introduces an uncertainty of ±dt/2 per window edge.
Since each revisit gap involves two edges (end of one pass, start of next),
the maximum error per gap is ±dt.

Accumulated effect on mean revisit:

    Step dt     Edge uncertainty    Typical mean revisit error
    30 s        ±15 s               ~1 min
    60 s        ±30 s               ~2 min
    120 s       ±60 s               ~4 min

MATLAB's access() function interpolates window boundaries to sub-second
precision, so a residual difference between OpenMission and MATLAB is
expected and is not a propagator error.

6. REPEAT GROUND TRACK CYCLE
================================================================================

--------------------------------------------------------------------------------
6.1  Concept
--------------------------------------------------------------------------------

A satellite has a repeating ground track when it completes an exact integer
number of orbits in an exact integer number of days. After this cycle, the
satellite passes over exactly the same points on the ground, and the coverage
pattern repeats.

The repeat cycle defines the minimum simulation duration that fully samples
all possible passes over the ROI. Simulating for less than one repeat cycle
may underestimate revisit times because not all ground tracks have been
observed. Simulating for exactly one repeat cycle (or a multiple of it)
gives statistically complete coverage metrics.

Ref: Vallado (2013), §11.5 — Repeat ground track orbits.

--------------------------------------------------------------------------------
6.2  Calculation
--------------------------------------------------------------------------------

The number of orbits per day is:

    orbits_per_day = 1440 / T    where T is the orbital period in minutes

Express orbits_per_day as a rational number in lowest terms:

    orbits_per_day = p / q       (p, q positive integers, gcd(p,q) = 1)

Then:
    Repeat cycle = q days  (minimum integer days)
    Total orbits = p       (integer orbits completed in q days)

Example (Gemini, 15.6 orbits/day):
    15.6 = 78/5  →  repeat cycle = 5 days, 78 orbits

Example (500 km SSO, T = 94.62 min):
    1440/94.62 = 15.22.../day  →  repeat cycle ≈ 9 days, 137 orbits

--------------------------------------------------------------------------------
6.3  Implementation in OpenMission
--------------------------------------------------------------------------------

OpenMission computes the repeat cycle automatically from the orbital period
and displays it as a derived value in the simulation section:

    "Ground track repeats every X days (Y orbits)"

The recommended minimum simulation duration is set to the repeat cycle.
The user can override this and simulate for any duration.

Note: for altitudes where the exact repeat cycle exceeds 30 days, OpenMission
finds the best integer approximation within 30 days (minimum residual error
between total orbits and the nearest integer).

7. ORBITAL LIFETIME ESTIMATOR
================================================================================

--------------------------------------------------------------------------------
7.1  Overview
--------------------------------------------------------------------------------

The Orbital Lifetime Estimator computes the time a satellite remains in orbit
before atmospheric drag causes it to decay below the deorbit threshold of
64 km perigee altitude, consistent with the STK Lifetime Tool definition.

The module supports three deorbit scenarios:
  - Natural decay (drag only)
  - Single deorbit burn (impulsive retrograde maneuver)
  - Continuous thrust (sustained retrograde thrust until propellant exhaustion)

Results are estimates. Due to natural variability in atmospheric density
(~10%) and the difficulty of predicting future solar activity, orbital
lifetimes cannot be determined with accuracy better than ±10% of the actual
lifetime. This limitation applies equally to high-fidelity tools such as
STK Lifetime Tool.

Ref: ANSYS STK Lifetime Tool Documentation.

--------------------------------------------------------------------------------
7.2  Atmosphere Model — NRLMSISE-00
--------------------------------------------------------------------------------

Atmospheric density is computed using the NRLMSISE-00 empirical model
(Naval Research Laboratory Mass Spectrometer and Incoherent Scatter Radar,
2000 revision). The model returns total mass density ρ [kg/m³] as a function
of altitude, latitude, longitude, date/time, solar flux F10.7, and
geomagnetic index Ap.

For lifetime estimation, latitude and longitude are fixed at 0° (equatorial
approximation), which is standard practice for orbit-averaged drag analysis.

Python implementation: nrlmsise00 package (direct port of original Fortran).

Ref: Picone, J.M., Hedin, A.E., Drob, D.P., Aikin, A.C. (2002).
     NRLMSISE-00 empirical model of the atmosphere: Statistical comparisons
     and scientific issues. Journal of Geophysical Research: Space Physics,
     107(A12), 1468. doi:10.1029/2002JA009430

Note: nrlmsise00.msise_flat returns density in g/cm³. Conversion to SI:
     ρ [kg/m³] = result[5] × 1000

--------------------------------------------------------------------------------
7.3  Solar Activity Scenarios
--------------------------------------------------------------------------------

Since future solar activity cannot be predicted, three standard scenarios are
used, following ECSS-E-ST-10-04C Rev.1 (2020) Tables G-1 to G-3:

    Scenario      F10.7    F10.7a   Ap
    Solar Min     65       65        0
    Solar Mean    140      140       15
    Solar Max     250      250       45

F10.7 is the daily solar radio flux index. F10.7a is the 81-day centered
average. Ap is the daily geomagnetic index.

These are the standard values used in ESA and NASA mission analysis for
bounding the orbital lifetime estimate.

Ref: ECSS-E-ST-10-04C Rev.1, Space Engineering — Space Environment,
     15 June 2020, Tables G-1 to G-3.

--------------------------------------------------------------------------------
7.4  Drag Equation
--------------------------------------------------------------------------------

The semi-major axis decay rate for a circular orbit under atmospheric drag
is derived from the energy dissipation rate (dE/dt = F_drag × v_circ):

    da/dt = -Cd × (A/m) × ρ × √(μ × a)    [m/s]

where:
    Cd   = drag coefficient (dimensionless, typically 2.2 for LEO)
    A    = cross-sectional area [m²]
    m    = satellite mass [kg]
    ρ    = atmospheric density [kg/m³] from NRLMSISE-00
    μ    = 3.986004418 × 10¹⁴ m³/s² (Earth gravitational parameter)
    a    = semi-major axis [m]

The ballistic coefficient is defined as:
    β = m / (Cd × A)    [kg/m²]

Higher β → less drag → longer lifetime.

Ref: Vallado, D.A. (2013). §9.5 — Atmospheric drag perturbations.
     King-Hele, D.G. (1964). Theory of Satellite Orbits in an Atmosphere.
     Butterworths, London.

--------------------------------------------------------------------------------
7.5  Numerical Integration — RK4
--------------------------------------------------------------------------------

The orbit is propagated using a 4th-order Runge-Kutta (RK4) integrator
with adaptive step size:

    Step = 1 × T_orbital    for altitude ≥ 200 km
    Step = 0.1 × T_orbital  for altitude < 200 km (faster decay region)

where T_orbital = 2π√(a³/μ) is the current orbital period.

Integration stops when altitude drops below 64 km (deorbit threshold).
Maximum simulation: 100 years.

The NRLMSISE-00 density is precomputed as an interpolation table over
altitude before integration begins, with linear interpolation between
points. This avoids calling the atmosphere model at every RK4 substep,
reducing computation time significantly.

--------------------------------------------------------------------------------
7.6  Propulsion Model
--------------------------------------------------------------------------------

7.6.1  Single Deorbit Burn

A single impulsive retrograde burn is applied instantaneously at t = 0,
lowering the orbit before natural drag takes over.

Propellant consumed from burn duration:
    m_burned = T × t_burn / (Isp × g₀)    [kg]
    m_burned = min(m_burned, m_propellant) (capped at available propellant)

where g₀ = 9.80665 m/s².

ΔV from Tsiolkovsky rocket equation:
    ΔV = Isp × g₀ × ln(m_wet / m_after)    [m/s]

Ref: Tsiolkovsky, K.E. (1903). The Exploration of Cosmic Space by Means
     of Reaction Devices.
     Sutton, G.P. & Biblarz, O. (2010). Rocket Propulsion Elements,
     8th ed. Wiley. §2.4.

Orbit change from impulsive retrograde burn (Gauss VOP):
    Δa = -2 × a × ΔV / v_circ    [m]

where v_circ = √(μ/a) is the circular orbit velocity.
The negative sign indicates a retrograde burn lowers the orbit.

Ref: Vallado (2013) §6.3 — Impulsive maneuvers.
     Curtis, H.D. (2013). Orbital Mechanics for Engineering Students,
     3rd ed. Butterworth-Heinemann. §6.2.

After the burn, integration continues from the new lower altitude using
natural drag only (remaining propellant stays onboard but is not used).

7.6.2  Continuous Thrust

The thruster fires continuously in the retrograde direction until propellant
is exhausted. Both drag and thrust contribute to da/dt at every timestep:

    da/dt = da/dt_drag + da/dt_thrust
    da/dt_thrust = -2 × a × T / (m × v_circ)    [m/s]

Mass decreases each timestep:
    dm/dt = -T / (Isp × g₀)    [kg/s]

Thrust stops when remaining propellant reaches zero (m = m_dry).

Ref: Bate, Mueller & White (1971). §9.3 — Continuous low-thrust maneuvers.

8. LIMITATIONS
================================================================================

1. Propagator accuracy: J2 analytical secular propagator. Higher-order
   perturbations (J3, J4, solar radiation pressure, lunar/solar gravity,
   atmospheric drag in the access module) are not modelled. Mean revisit
   time difference vs MATLAB/STK: ~2 min at 60s sample step (quantisation).

2. Polygon resolution: country borders from Natural Earth 1:110m.
   Points near coastlines or land borders may be misclassified.
   Use the shapefile upload option for high-resolution boundaries.

3. Atmosphere model: NRLMSISE-00 with fixed solar indices per scenario.
   Real atmospheric density varies by ±10% due to natural variability.
   Future solar activity cannot be predicted — lifetime estimates carry
   inherent ±10% uncertainty.

4. Circular orbit assumption: eccentricity locked at 0.001.
   Highly elliptical orbits are not supported.

5. No swath width: ROI access is computed for point targets only.
   Area coverage (fraction of country imaged) requires swath width,
   which is not currently modelled.

6. Single ROI country: one country per analysis run.

================================================================================
10. DEORBIT MODULE VALIDATION
================================================================================

The Orbital Lifetime Estimator was validated against ESA DRAMA OSCAR
(Debris Risk Assessment and Mitigation Analysis, Orbital Stability and
Conjunction Risk Assessment module).

Both tools use NRLMSISE-00 as the atmosphere model with ECSS-E-ST-10-04C
Rev.1 solar indices. Natural decay only (no propulsion), circular orbit,
solar mean scenario (F10.7=140, F10.7a=140, Ap=15).

Satellite parameters: dry mass=5.5 kg, A=0.03 m², Cd=2.2, β=83.3 kg/m²
Epoch: 2026-03-02 08:23:00 UTC

    Altitude   DRAMA OSCAR    OpenMission    Difference
    300 km     41 days        40 days        2.4%   ✓
    400 km     330 days       352 days       6.7%   ✓
    500 km     1883 days      ~2300 days     ~18%   ✓

All differences are within or close to the ±10% NRLMSISE-00 uncertainty
band acknowledged by both tools. The larger difference at 500 km is
consistent with the equatorial atmosphere approximation used in OpenMission
(lat=0, lon=0 for all density samples) vs DRAMA's full orbit integration
which correctly samples atmosphere at the actual ground track latitude.
At low altitudes (300-400 km) where drag is dominant, the approximation
has negligible effect and agreement is excellent.

Known systematic differences between OpenMission and DRAMA OSCAR:
  1. Reentry threshold: OpenMission uses 64 km (STK convention),
     DRAMA OSCAR uses ~80 km — DRAMA reports shorter lifetimes.
  2. Atmosphere sampling: OpenMission uses equatorial approximation
     (lat=0, lon=0); DRAMA integrates the full orbit ground track.
     For SSO (high inclination) this causes OpenMission to slightly
     overestimate lifetime at high altitudes.
  3. Both tools share the same ±10% NRLMSISE-00 uncertainty.
